# Button

A single clickable action with five variants (`primary`, `secondary`, `ghost`, `danger`, `link`),
three sizes, and disabled/loading states that are visually and behaviorally distinct. `link`
renders as bare inline text — no padding, border, or background — for secondary actions that
should read as a text link rather than a button (e.g. "Back to sign in").

## User story

**US-CW-023 — Component library — atoms & foundations**

As a clearline-web engineer, I want StatusBadge/MoneyDisplay/BudgetGauge and the 8 Tier-1 atoms
available as typed React components, so that I never hand-roll a button, badge, or money value
and risk missing an accessibility or formatting rule the design system already solved.

## Acceptance criteria

- AC-01 — AC-04 — StatusBadge/MoneyDisplay/BudgetGauge, covered on their own docs pages.
- AC-05 — Checkbox wraps Radix `Checkbox` for correct `onCheckedChange`/`aria-checked` behavior.
- **AC-06 — Button.** Given Button with `loading={true}` or `disabled={true}`, when rendered, then
  the button stays focusable (`aria-disabled` — and, for `loading`, `aria-busy` — are set rather
  than the native `disabled` attribute, so keyboard/screen-reader focus isn't lost on a
  not-yet-submittable or in-flight form), and does not fire `onClick` or submit an enclosing form.
- AC-07 — TextField's `state="error"` sets `aria-invalid` and `aria-describedby`.

### Edge case covered here

Disabled and loading are both exercised as separate stories — both block `onClick` via the same
`aria-disabled` + guarded-click-handler mechanism and stay focusable, but only loading shows the
spinner and `aria-busy`, so the two states must never be conflated.
