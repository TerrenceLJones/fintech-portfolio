# AppShell

A layout-route component: mount it on a parent `<Route>` and nest page routes under it, so each
page renders into its `<Outlet/>` with zero per-page chrome-wrapping. Lays the app out as a
persistent left **sidebar rail** — brand, role-scoped vertical navigation, and a footer holding the
relocated Light/Dark control and the user-identity block (avatar · name · role · approval limit) —
beside a scrollable content column (design §3.1). On narrow viewports the rail collapses behind a
hamburger into an off-canvas drawer (EPIC-CW-021 / US-CW-032). Assumes a `<ThemeProvider>` ancestor
mounted once at the app root, rather than owning one itself.

## User story

**US-CW-025 — Component library — organisms & decorators**

As a clearline-web engineer, I want DataTable/BulkActionResult/EmptyState and
the AppShell decorators available as typed React components, so that I
can assemble a full screen (selection, sorting, approval flows, partial-failure summaries) by
composing already-built pieces instead of writing table selection/sort logic or approval-limit
disabling logic from scratch in each feature epic.

## Acceptance criteria

- AC-01 — AC-02 — DataTable's select-all toggle and vendor-column sort.
- AC-04 — BulkActionResult excludes failures from the committed count and lists each reason.
- **AC-05 — AppShell.** Given `<AppShell>` wrapping any page content, with a `<ThemeProvider>`
  mounted once at the app root, when rendered, then it renders the sidebar rail with a Light/Dark
  toggle (relocated from the old top bar into the rail footer) backed by that shared theme state, and
  requires zero additional per-page token setup.

### Edge case covered here

The theme toggle affects the whole subtree via `data-theme` on `document.documentElement`, not
just component-local state — verified against the document element itself, not a prop.
