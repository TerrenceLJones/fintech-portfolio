# BulkActionResult

The result of a bulk approve/reject, in two shapes with one shared guarantee — successes stay
committed. A **partial failure** keeps each failure visible with its per-row reason and a
retry-failed-only action (never an all-or-nothing rollback). A **mid-batch connection drop**
(pass `notProcessed`) switches to a negative banner naming how many were confirmed before the
drop, with the confirmed and "not yet processed" items split into two columns and a resume that
retries only the unprocessed tail.

## User story

**US-CW-025 — Component library — organisms & decorators**

As a clearline-web engineer, I want DataTable/BulkActionResult/EmptyState and
the AppShell decorators available as typed React components, so that I
can assemble a full screen (selection, sorting, approval flows, partial-failure summaries) by
composing already-built pieces instead of writing table selection/sort logic or approval-limit
disabling logic from scratch in each feature epic.

## Acceptance criteria

- AC-01 — AC-02 — DataTable's select-all toggle and vendor-column sort.
- **AC-04 — BulkActionResult.** Given BulkActionResult with `total=10` and 2 `failures`, when it
  renders, then it shows "8 of 10 approved" (the committed count excludes the failures) with each
  failure's specific reason listed, plus a "Retry failed (2)" action.
- AC-05 — AppShell owns theme state and requires zero additional per-page token setup.

**US-CW-013 — Batch Approval & Bulk Actions**

- **AC-03 — mid-batch network drop.** Given a batch interrupted after some items were confirmed,
  when it renders with `notProcessed`, then the confirmed items are kept, the rest are shown as
  "not yet processed", and "Retry N unprocessed" resumes only those (see the `NetworkDrop` story).

### Edge case covered here

Zero failures still renders correctly — no failure rows, no crash — as its own story rather than
only the partial-failure case.
